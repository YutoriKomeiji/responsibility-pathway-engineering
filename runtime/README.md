# Runtime scaffold

Status: bounded runtime-facing scaffold / Open Construction.

## Purpose

This directory is a small runtime-facing surface for Responsibility Pathway Engineering.

It is intended to explain how future runtime-adjacent examples may connect responsibility-path records to execution-adjacent workflows without claiming that this repository provides a production runtime, service, SDK, connector, or autonomous execution system.

## What this scaffold is

This scaffold may be used to organize future examples that show how a responsibility-path record could sit near an action, tool call, review flow, automation, or execution request.

The runtime-facing question is:

> Before, during, or after an execution-adjacent step, can the responsibility path remain visible, bounded, stoppable, returnable, and repairable?

## Initial scope

The first runtime-facing layer is documentation only.

It may describe:

- where a responsibility-path record could be attached
- what evidence should be preserved
- what uncertainty should remain visible
- where approval gates are expected
- where stop authority remains
- where repair ownership returns
- which claims are excluded
- which checker or formalization surfaces may later inspect the record

## Human-maintainer boundary

Human maintainer judgment remains required for:

- runtime design direction
- example acceptance
- merge decisions
- publication decisions
- connector decisions
- release decisions
- external claims
- final responsibility

A runtime-facing artifact may assist review, but it does not become final approval.

## Relationship to examples

Runtime-facing examples should start small.

A good first example should be:

- local
- synthetic
- copyable
- reviewable by a human
- bounded by explicit excluded claims
- compatible with existing responsibility-path examples
- clear that it does not execute real-world actions
