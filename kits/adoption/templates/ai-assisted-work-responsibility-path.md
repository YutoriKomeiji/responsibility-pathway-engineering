# AI-Assisted Work Responsibility Path Template Pointer

This file points first users of the RPE Adoption Kit to the current source template:

- `templates/ai-assisted-work-responsibility-path.yaml`

This pointer is intentionally not a copy of the YAML template.

Keeping this file as a pointer avoids silent drift between a kit-local copy and the source template while the adoption kit is still under construction.

## Use order

Before using the source template, read:

1. `kits/adoption/README.md`
2. `kits/adoption/non-claims.md`
3. `kits/adoption/quickstart.md`
4. `templates/ai-assisted-work-responsibility-path.yaml`

The README explains the kit entrance.

The non-claims file explains what the kit, templates, examples, checkers, forks, and formalization do not prove.

The quickstart explains the first-use path for one bounded AI-assisted work case.

The source YAML template provides the current copyable responsibility-path structure.

## What this pointer enables

This pointer helps a first user find the current source template without treating the adoption kit as a separate template authority.

It supports:

- first-use navigation
- source-reference discipline
- drift avoidance
- fork safety
- template boundary preservation

## What this pointer does not provide

This pointer does not provide a new template.

It does not certify, validate, approve, or prove any use of the source template.

It does not provide:

- legal review
- safety review
- compliance review
- fairness review
- production approval
- conformance evidence
- runtime correctness proof
- adapter correctness proof
- connector correctness proof
- semantic responsibility proof
- AI final-responsibility transfer

## Source-reference rule

Until the adoption kit deliberately introduces a kit-local YAML copy, the source template remains:

- `templates/ai-assisted-work-responsibility-path.yaml`

If a future PR copies that template into `kits/adoption/templates/`, that PR should also add drift notes explaining:

- why a copy is needed
- what source file it was copied from
- what differences exist, if any
- how future drift will be detected or reviewed
- what a copied template still does not prove

## How to use the source template safely

When using the source template, fill only one bounded AI-assisted work case first.

Record at minimum:

- the work item
- the responsible human or institution
- where AI assisted
- what AI did not decide
- checked evidence
- unchecked sources
- missing context
- review gate
- approval status
- return, repair, dispute, or reopening path
- non-claims

Stop before external use if any of these are unclear.

## RPE function

Inside Responsibility Pathway Engineering, this pointer supports **template route stabilization**.

Its role is not to define a new record schema.

Its role is to keep the adoption kit connected to the current source template while preserving:

- responsibility localization
- AI assistance boundary
- evidence and missing-context visibility
- review and approval separation
- return-path construction
- non-claim preservation
- fork safety
